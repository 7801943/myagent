# 多用户支持实施方案

## 1. 目标与结论

本方案用于在当前 `多用户支持` 分支上，将 MyAgent 从“单用户 Web + 全局会话/状态/工作区”升级为“多用户登录、隔离会话、私有/公共工作区、按用户配置控制工具与 Skill 可见性，并支持公共目录 OnlyOffice 协同编辑”的架构。

核心设计原则：

- 用户身份必须贯穿认证、WebSocket、REST API、SessionManager、StateStore、Workspace、工具执行、Skill 加载、OnlyOffice token 和回调保存。
- 会话和状态数据库按用户隔离，每个用户一个全新的 SQLite 数据库，不复用旧的全局 `myagent_state.db`。
- 工作区采用统一虚拟路径模型：`private/...` 表示用户私有目录，`public/...` 表示公共目录。
- 工具和 Skill 文件本身是公共资源，但通过用户配置决定可见性，并在执行层强制拒绝不可见工具。
- 公共目录对 agent 是只读工作区。agent 可以读取公共目录内容，但不得通过任何工具写入或修改公共目录文件。
- 公共目录文档允许用户手动上传，并允许通过 OnlyOffice 协同编辑保存。

## 2. 当前代码审查结论

当前代码已经具备一些可复用基础，但多用户隔离边界还不完整：

- `AuthService` 已支持从 `data/users.json` 加载多个账号，但用户模型只有 `username/password_hash/active_ips`，没有 group、可见工具、可见 Skill 或用户工作区配置。
- `myagent/interfaces/web/dependencies.py` 中目前只初始化一个全局 `SQLiteStateStore()` 和一个全局 `SessionManager`，所有用户共享一个状态数据库。
- `SessionManager` 的 `_sessions` 使用 `session_id` 作为唯一键；`join_session/list_sessions/delete_session/get_session_messages` 没有强制用户边界，存在跨用户会话可见、恢复和删除风险。
- `workspace_files` 路由目前通过请求体里的 `session_id` 直接查内存 session，没有校验当前 token 用户是否拥有该 session。
- `WorkspaceManager` 目前是单 root 状态容器，不支持私有目录和公共目录两类物理根。
- `DocumentService` 当前按单个 `workspace_root` 解析路径，OnlyOffice token 没有包含 workspace scope，也没有区分公共目录写入来源。
- Skill 当前按 `prompts/skills/<username>/...` 加载，和“Skill 公用、按用户配置可见”目标相反。
- 内置文件工具只做敏感系统路径黑名单检查，不理解 MyAgent 的 `private/public` 工作区权限。

## 3. 默认决策

如果没有额外产品要求，第一版按以下默认落地：

- 用户和权限先继续使用配置文件管理，不做前端用户管理页面。
- 只有 `admin` 账号属于 `admin` 组；其他用户属于普通 `user` 组。
- `admin` 默认可见全部工具和全部 Skill。
- 普通用户默认也可见全部工具和 Skill，但可以在用户配置里收窄。
- 普通用户可上传公共目录文件；公共目录删除和重命名仅 `admin` 可做。
- 私有目录保留用户自己的上传、删除、重命名、OnlyOffice 预览和编辑能力。
- 公共目录支持 OnlyOffice 多用户协同编辑。
- agent 读取公共目录允许；agent 写入、修改、重命名、删除公共目录一律拒绝。
- `full_access` 安全策略不能绕过公共目录写保护。
- 旧的 `myagent_state.db` 不迁移。升级后每个用户从新的 per-user DB 开始。

## 4. 用户与配置模型

### 4.1 `data/users.json` 新结构

扩展 `data/users.json` 为 schema v2：

```json
{
  "schema_version": 2,
  "users": [
    {
      "username": "admin",
      "password_hash": "pbkdf2_sha256$...",
      "group": "admin",
      "disabled": false,
      "active_ips": [],
      "visible_tools": ["*"],
      "visible_skills": ["*"],
      "workspace": {
        "private_dir": null
      }
    },
    {
      "username": "alice",
      "password_hash": "pbkdf2_sha256$...",
      "group": "user",
      "disabled": false,
      "active_ips": [],
      "visible_tools": ["file_read", "file_query", "file_diff", "file_write", "file_edit", "file_edit_table", "cli_execute", "use_skill"],
      "visible_skills": ["office-automation"],
      "workspace": {
        "private_dir": null
      }
    }
  ]
}
```

兼容规则：

- 缺少 `schema_version` 时按旧格式读取。
- 缺少 `group` 时，`username == "admin"` 自动为 `admin`，其他用户自动为 `user`。
- 缺少 `disabled` 时默认为 `false`。
- 缺少 `visible_tools` 或 `visible_skills` 时默认为 `["*"]`。
- 缺少 `workspace.private_dir` 时使用全局 private dir template。

### 4.2 新增配置项

建议在 `config.yaml.example` 增加顶层配置：

```yaml
multi_user:
  enabled: true

state:
  per_user_dir: "data/state/users"

workspace:
  base_dir: "workspaces"
  private_dir_template: "workspaces/users/{username}"
  public_dir: "workspaces/public"
  public_upload_enabled: true
  public_rename_delete_group: "admin"

skills:
  common_dir: "prompts/skills/common"
```

说明：

- `state.per_user_dir` 是每个用户 SQLite 的父目录。
- `workspace.private_dir_template` 只接受 `{username}` 占位符，用户名必须经过安全归一化。
- `workspace.public_dir` 是所有用户共享目录。
- `skills.common_dir` 是公共 Skill 根目录。现有 `prompts/skills/admin/office-automation` 应迁移到 `prompts/skills/common/office-automation`。

## 5. 认证与用户上下文

### 5.1 后端模型

在 `myagent/interfaces/web/auth.py` 中扩展：

- `UserData`
  - `username`
  - `password_hash`
  - `group`
  - `disabled`
  - `active_ips`
  - `visible_tools`
  - `visible_skills`
  - `workspace`

- `TokenInfo`
  - `token`
  - `username`
  - `group`
  - `ip`
  - `created_at`
  - `visible_tools`
  - `visible_skills`
  - `workspace`

新增帮助方法：

- `AuthService.get_user_profile(username) -> UserData | None`
- `AuthService.is_admin(username) -> bool`
- `AuthService.visible_tools(username) -> set[str] | Literal["*"]`
- `AuthService.visible_skills(username) -> set[str] | Literal["*"]`

登录逻辑调整：

- 如果用户 `disabled == true`，登录失败。
- token 中保存 group 和 visibility 快照。
- `/api/auth/me` 返回 `username/group/visible_tools/visible_skills` 摘要。

### 5.2 WebSocket token 生命周期

当前 WebSocket 断开时会调用 `auth_service.logout(token)`，这会导致浏览器刷新或短暂断线后 HTTP token 也失效。多用户版本建议调整：

- WebSocket 断开只 detach client，不自动 logout token。
- 主动点击登出时才调用 `/api/auth/logout`。
- token 过期仍由 `token_ttl_seconds` 控制。
- IP 绑定仍在登录和主动登出时维护。

## 6. 每用户数据库与 Session 隔离

### 6.1 新增 `UserStateStoreRegistry`

新增模块建议放在 `myagent/context/user_state.py` 或 `myagent/interfaces/web/services/state_registry.py`。

职责：

- 根据用户名生成安全目录名。
- 懒加载并初始化该用户的 `SQLiteStateStore`。
- 管理所有已打开的 SQLite 连接。
- shutdown 时关闭全部 store。

路径规则：

```text
data/state/users/admin/myagent_state.db
data/state/users/alice/myagent_state.db
```

目录名归一化：

- 只保留字母、数字、下划线、短横线。
- 其他字符替换为 `_`。
- 空值、`.`、`..` 一律拒绝或回退为安全名。

### 6.2 改造 `SessionManager`

当前 `SessionManager` 构造时接收单个 `state_store`。多用户版本改为：

- Web 场景传入 `state_store_registry`。
- CLI 场景仍可传单个 `state_store` 或不传，保持兼容。

内存 session key 改为：

```python
SessionKey = tuple[str, str]  # (username, session_id)
self._sessions: dict[SessionKey, Session]
```

方法签名调整：

- `create_session(user, ...)`
  - 从 registry 取当前用户 store。
  - workspace root 使用新的虚拟 workspace，不再直接使用 `agent.root_dir`。

- `get_session(user, session_id)`
  - 只查 `(user.username, session_id)`。

- `join_session(user, session_id=None, ...)`
  - 如果指定 session id，只能恢复当前用户 DB 中的 session。
  - 不存在时返回 404 或创建新会话，具体按调用场景区分。

- `list_sessions(user)`
  - 只列当前用户 DB。

- `delete_session(user, session_id)`
  - 只删除当前用户内存 session 和当前用户 DB 中的记录。

- `get_session_messages(user, session_id)`
  - 只读当前用户 DB。

### 6.3 WebSocket 会话操作

`ws_handler.py` 中所有 session 操作必须使用当前 token 用户：

- `session_list` 调 `list_sessions(user)`。
- `session_switch` 调 `join_session(user, target_id)`。
- `session_delete` 调 `delete_session(user, target_id)`。
- `session_create` 调 `create_session(user, new_session_id)`。

会话列表标题和消息数的读取也必须传 user，避免通过 session id 探测其他用户历史。

## 7. Workspace 双目录模型

### 7.1 虚拟路径

所有 Web、LLM prompt、文件树、OnlyOffice 都统一使用虚拟相对路径：

```text
private/report.docx
private/project/a.py
public/shared/report.docx
public/team/table.xlsx
```

用户看到的路径和 agent prompt 中的路径都应该是这种形式。

### 7.2 新增 `WorkspaceResolver`

建议新增 `myagent/core/workspace_resolver.py`。

核心类型：

```python
WorkspaceArea = Literal["private", "public"]
WorkspaceActor = Literal["user", "agent", "onlyoffice", "admin"]
WorkspaceOperation = Literal["read", "write", "upload", "rename", "delete", "list"]
```

`WorkspaceResolver` 初始化参数：

- `username`
- `group`
- `private_root`
- `public_root`

核心方法：

- `resolve(virtual_path, operation, actor) -> ResolvedWorkspacePath`
- `resolve_existing_file(virtual_path, operation, actor)`
- `scan(area, sub_path=None) -> list[FileInfo]`
- `can(actor, operation, virtual_path) -> bool`
- `to_virtual_path(real_path) -> str | None`
- `is_under_public(real_path) -> bool`
- `is_under_private(real_path) -> bool`

权限规则：

| actor | private read | private write | public read | public write |
| --- | --- | --- | --- | --- |
| user | yes | yes | yes | upload yes, rename/delete admin only |
| admin | yes for own private | yes for own private | yes | yes |
| agent | yes | yes | yes | no |
| onlyoffice | yes | yes | yes | yes |

补充规则：

- `agent` 对 public 的 `write/rename/delete/upload` 永远拒绝。
- `onlyoffice` 只能在 token 证明 `mode=edit` 时写。
- `user` 上传 public 允许，但 public 删除/重命名仅 admin。
- 任何 actor 都不能通过 `..`、绝对路径、Windows drive、符号链接越界。

### 7.3 改造 `WorkspaceManager`

`WorkspaceManager` 保持“状态容器，不直接做文件 I/O”的原则，但状态结构扩展：

`FileInfo` 新增：

- `area`: `"private" | "public"`
- `can_read`
- `can_upload`
- `can_rename`
- `can_delete`
- `can_agent_read`
- `can_agent_write`

`WorkspaceState.root_path` 从物理路径改为虚拟展示名，例如：

```text
workspace://<username>
```

文件树初始扫描返回两个顶层目录：

```text
private/
public/
```

`scan_dir`、`files_changed` 通过 resolver 完成真实目录扫描，再写回虚拟路径。

## 8. REST API 改造

### 8.1 Session REST

`myagent/interfaces/web/routes/sessions.py`：

- `list_sessions(request)` 从 `request.state.user` 取当前用户。
- `delete_session(session_id, request)` 只删除当前用户 session。
- 返回 403/404 时避免泄露“这个 session id 是否属于别人”。

### 8.2 Workspace Files REST

`myagent/interfaces/web/routes/workspace_files.py`：

- 所有端点增加 `Request` 参数。
- `_session_for_request(request, session_id)` 必须校验：
  - token 有效。
  - session 存在。
  - session.user.username == request.state.user.username。

端点行为：

- `preflight`
  - 支持 `target_dir` 为 `private/...` 或 `public/...`。
  - 调 resolver 校验 upload 权限。

- `upload`
  - 普通用户可上传到 private 和 public。
  - public 上传允许覆盖，但必须经过现有 preflight 冲突确认。
  - 保存后发 `workspace.files_changed`，路径使用虚拟路径。

- `rename`
  - private 允许当前用户。
  - public 仅 admin。

- `delete`
  - private 允许当前用户。
  - public 仅 admin。

- `download`
  - private 和 public 都允许当前用户下载。

### 8.3 Documents REST

`myagent/interfaces/web/routes/documents.py`：

- `editor-config`
  - 从当前用户和 session 获取 resolver。
  - `path` 必须是虚拟路径。
  - public 文档 mode 默认 edit，PDF 强制 view。

- `download`
  - OnlyOffice DocumentServer 使用文档 access token 下载。
  - token 必须包含 path、scope、username、mode、session_id。

- `callback`
  - 验 token。
  - token 的 `mode` 必须是 `edit`。
  - `actor=onlyoffice` 调 resolver 保存。
  - 保存成功后广播对应 workspace 更新：
    - private：通知该用户相关活跃 session。
    - public：通知所有活跃 session。

## 9. 工具与 Skill 可见性

### 9.1 公共工具，按用户过滤

工具仍从公共目录注册：

- 内置工具：`file_read/file_query/file_diff/file_write/file_edit/file_edit_table/cli_execute/use_skill`
- 热加载工具：`myagent/tools/tools_store`
- MCP 工具：保持现有注册机制

新增 `ToolVisibilityPolicy`：

- 输入：当前用户 `visible_tools`。
- schema 层：`ToolManager.list_schemas()` 后过滤，只把可见工具交给 LLM 和前端。
- 执行层：`ToolInterface.execute()` 开始时先判断工具是否可见，不可见直接返回 error。
- `visible_tools=["*"]` 表示全部可见。

### 9.2 公共 Skill，按用户过滤

Skill 目录调整：

```text
prompts/skills/common/office-automation/skill.md
```

`SkillRegistry` 改为：

- `load_from_common_dir(common_dir, active_names=None, visible_names=None)`
- `visible_skills=["*"]` 时加载全部。
- `visible_skills=["office-automation"]` 时只加载匹配项。

`use_skill` 工具也必须使用过滤后的 registry，因此不可见 Skill 无法被加载。

现有测试 `test_skill_registry_loads_only_requested_username` 应改为验证 common Skill 加载和用户 visibility 过滤。

## 10. Agent 工具公共目录写保护

### 10.1 `WorkspacePermissionPolicy`

新增执行前策略，挂到 `ToolInterface.execute()`，并且优先级高于现有 safety policy。

处理规则：

- 对 `file_read/file_query/file_diff`
  - 将虚拟路径或合法绝对路径解析为真实路径。
  - public 和 private 都可读。

- 对 `file_write/file_edit/file_edit_table`
  - 解析 `path`。
  - 如果 path 在 public，直接拒绝。
  - 如果 path 在 private，替换成真实路径后继续执行。

- 对 `cli_execute`
  - `cwd` 为空时默认设置为当前用户 private root。
  - 如果 `cwd` 在 public，直接拒绝。
  - 如果 command 文本包含 public 虚拟路径或 public 真实路径，直接拒绝。
  - 不通过 CLI 操作 public，即使是只读命令也不鼓励；public 读取应使用 `file_read/file_query/file_diff`。

### 10.2 路径重写

为了让 LLM 和前端都能稳定使用虚拟路径，ToolInterface 在执行前做路径重写：

- `private/a.txt` -> `/abs/workspaces/users/alice/a.txt`
- `public/b.docx` -> `/abs/workspaces/public/b.docx`
- 绝对路径如果位于当前用户 private 或 public 下，也能识别并套用权限。
- 普通相对路径没有 `private/` 或 `public/` 前缀时，按 `private/` 处理，保持旧提示兼容。

### 10.3 Prompt 提示

动态 system prompt 中加入工作区规则：

- 当前用户私有目录：`private/`
- 公共只读目录：`public/`
- 读取公共文件可用 `file_read`。
- 不允许使用任何工具写入、编辑、删除或移动 `public/` 文件。
- OnlyOffice 是公共目录文档的唯一协同编辑通道。

## 11. OnlyOffice 协同设计

### 11.1 Editor config

`DocumentService.build_editor_config()` 改造参数：

- `virtual_path`
- `username`
- `group`
- `mode`
- `session_id`
- `resolver`

返回 config 时：

- `document.title` 使用真实文件名。
- `document.key` 对 public 文档在所有用户间一致。
- `editorConfig.user.id` 使用 username。
- `editorConfig.user.name` 使用 username。
- `permissions.edit` 由 resolver 和 mode 决定。

### 11.2 Token

文档 access token payload：

```json
{
  "path": "public/docs/report.docx",
  "scope": "public",
  "username": "alice",
  "group": "user",
  "mode": "edit",
  "session_id": "abc123",
  "iat": 123,
  "exp": 456,
  "nonce": "..."
}
```

校验规则：

- path 必须完全匹配。
- scope 必须与 resolver 解析结果一致。
- callback 保存必须 `mode == "edit"`。
- token 过期拒绝。
- 如果启用 OnlyOffice JWT，继续使用现有 `onlyoffice_jwt_secret`。

### 11.3 Callback 保存

OnlyOffice callback status `2` 或 `6` 时：

1. 校验 access token。
2. 用 resolver 解析 path，actor 为 `onlyoffice`，operation 为 `write`。
3. 拉取 OnlyOffice 提供的临时下载 URL。
4. 原子写入目标文件。
5. 返回 `{"error": 0}`。
6. 触发 workspace 刷新广播。

公共目录协同效果：

- 多个用户打开同一个 `public/...` 文档时，OnlyOffice DocumentServer 按相同 document key 进入同一协同编辑会话。
- 保存后，所有活跃用户的 public 文件树刷新。
- agent 不能通过工具改 public，但用户可通过 OnlyOffice 修改 public。

## 12. 前端改造

### 12.1 登录态

`web/js/auth.js`：

- 保存 `group` 和权限摘要。
- `validateToken()` 成功后更新 `state.authUser`：

```js
state.authUser = {
  username: "alice",
  group: "user",
  visible_tools: ["*"],
  visible_skills: ["office-automation"]
};
```

### 12.2 Workspace 文件树

`web/js/workspace.js`：

- 文件树根显示 `private` 和 `public`。
- `openWorkspaceFile()` 发送虚拟路径。
- 上传默认目标：
  - 在 private 节点上右键，默认 `private/...`
  - 在 public 节点上右键，默认 `public/...`
- 右键菜单根据后端能力字段禁用：
  - `can_rename`
  - `can_delete`
  - `can_upload`

公共目录菜单：

- 普通用户可见：上传、下载、复制相对路径、预览/编辑、ai读取。
- 普通用户不可用：重命名、删除。
- admin 可用：重命名、删除。

### 12.3 OnlyOffice

`web/js/onlyoffice-editor.js`：

- 继续把 `state.currentSessionId` 传给 `/api/documents/editor-config`。
- `path` 传虚拟路径。
- 不在前端判断真实目录，所有权限以后端为准。

## 13. 推荐实施顺序

### 阶段 1：身份与 per-user DB

- 扩展 `AuthService` 用户模型。
- 扩展 `/api/auth/me`。
- 新增 `UserStateStoreRegistry`。
- 改造 `dependencies.py` 初始化。
- 改造 `SessionManager` 的 store 获取和 session key。
- 改造 WebSocket session list/create/switch/delete。
- 改造 REST sessions 路由。
- 测试：两个用户会话完全隔离。

## 15. 验收标准

功能验收：

- admin 和普通用户可同时登录。
- 用户只能看到自己的会话列表。
- 用户无法通过猜测 session_id 获取或删除他人会话。
- 每个用户有独立 DB 文件。
- 文件树显示 `private/` 和 `public/`。
- 普通用户可上传公共文档。
- 普通用户不可删除或重命名公共文档。
- admin 可删除和重命名公共文档。
- agent 可以读取 public 文档。
- agent 使用 `file_write/file_edit/file_edit_table/cli_execute` 修改 public 时被拒绝。
- 用户可通过 OnlyOffice 打开并协同编辑 public 文档。
- OnlyOffice 保存 public 文档后，所有相关客户端刷新文件状态。
- 用户配置隐藏某工具或 Skill 后，前端状态、LLM schema 和执行层都不可见或不可用。

安全验收：

- `full_access` 策略不能绕过 public 写保护。
- REST API 不信任前端传来的 `session_id`，必须结合 token 用户校验。
- OnlyOffice callback 不依赖浏览器 token，只依赖短期文档 token，并校验 path/scope/mode/exp。
- 所有真实路径解析都必须经过 resolver，禁止路径越界。

## 16. 风险与注意事项

- `cli_execute` 很难在纯 subprocess 模式下做到文件系统级强隔离。第一版必须从执行策略上禁止 CLI 触碰 public 路径；更强隔离可在后续用 Docker 或 OS 权限把 public 以只读方式挂载给 agent 工具进程。
- 旧全局数据库不迁移会导致升级后看不到旧会话，这是本方案的明确默认。如果需要迁移，应单独做迁移脚本，按目标用户导入。
- Skill 目录从用户私有改为 common 后，现有测试和路径说明都要同步更新。
- 公共目录允许普通用户上传，意味着需要实际部署时考虑磁盘配额、文件大小和审计日志。当前第一版只复用现有上传限制和压缩包拒绝策略。
- OnlyOffice DocumentServer 的 public URL、internal URL、JWT secret 必须配置正确，否则公共协同编辑会表现为能打开但无法保存或无法加载 API。
