## 二、Session 模块接管状态同步

### 2.1 状态枚举精简

```python
class AgentState(str, Enum):
    IDLE = "idle"                    # 空闲 / run() 结束后回归此状态
    THINKING = "thinking"            # LLM 推理阶段
    GENERATING = "generating"        # LLM 流式输出阶段（替代 RUNNING）
    WAITING_TOOL = "waiting_tool"    # 工具批量执行中
    WAITING_HITL = "waiting_hitl"    # 人工审批（Phase 2 预留）
    ERROR = "error"                  # 发生错误
```

**变化**：
- ❌ 移除 `FINISHED` — 正常结束后回到 `IDLE`
- 🏷 `RUNNING` → `GENERATING`（更精准）

### 2.2 状态流转图

```
IDLE ──run()开始──▶ THINKING
                       │
                  (首 chunk 到达)
                       ▼
                   GENERATING
                       │
              ┌────────┴────────┐
              ▼                 ▼
       WAITING_TOOL          IDLE (无工具，结束)
              │
      (工具全部执行完)
              ▼
          THINKING (进入下一轮)
              
异常路径：任意阶段 → ERROR
```

### 2.3 Turn → Session 状态同步（Hook 桥接）

**原则**：Turn 不持有 Session。Agent 注册一个 hook 回调即可桥接两者：

```python
# agent.py — __init__ 末尾新增
self._hooks.on("state_change", self._on_session_state_change)

async def _on_session_state_change(self, ctx, state):
    session = self._session_manager.active
    if session:
        try:
            session.agent_state = AgentState(state)
        except ValueError:
            pass  # 未知状态值忽略
```

Turn 层 `state_change` 事件照常 emit，无需感知 Session 存在。

## 三、持久化逻辑从 Agent → Session

### 3.1 Session 新增方法

```python
# session.py
from myagent.context.message import Message

class Session:
    async def persist(self, messages=None, state=None, metadata=None):
        """一站式：更新内存状态 + 写持久层"""
        if state is not None:
            self.agent_state = state
        if metadata is not None:
            self.metadata.update(metadata)
        if messages is not None:
            await self.save(messages)
        else:
            await self.save()

    async def load_messages(self) -> list[Message]:
        """加载该会话的全部消息历史"""
        if not self._state_store:
            return []
        return await self._state_store.load_messages(self.id)
```

### 3.2 Agent.run() 改前 vs 改后

**改前（4 个分支各有 3~4 行直接调 StateStore）**：
```python
if self._state_store:
    await self._state_store.save_messages(self._session_id, self._context.messages)
    await self._state_store.save_state(self._session_id, AgentState.FINISHED, {...})
```

**改后（提取 + 统一走 Session）**：
```python
async def _persist(self, state=None, metadata=None):
    session = self._session_manager.active
    if session:
        await session.persist(self._context.messages, state, metadata)

# 各分支只需一行：
await self._persist(AgentState.IDLE, {"stop_reason": "completed"})          # 正常
await self._persist(AgentState.IDLE, {"cancelled": True, "cancel_reason": ...})  # 取消
await self._persist(AgentState.ERROR)                                        # 异常
```

### 3.3 Agent.restore_session 改前 vs 改后

```python
# 改前
messages = await self._state_store.load_messages(session_id)  # 直调

# 改后  
messages = await session.load_messages()                      # 走 Session
```

结果：Agent 中 `self._state_store` 引用从 5 处 → **0 处**。

## 七、补充：设计思路与实施注意事项

> 以下内容供编码 agent 深入理解设计意图，避免机械执行。

### 7.1 架构设计原则

**原则 1：谁持有数据谁负责持久化**
- `Session` 持有 `session_id`、`agent_state`、`metadata`、`_state_store`
- 消息历史是会话的子数据
- 所以 `Session` 应该提供 `persist()` 和 `load_messages()`
- Agent 不应绕过 Session 直接调 StateStore

**原则 2：生产者和消费者通过 Hook 解耦**
- Turn 是状态变化的生产者（emit `state_change`）
- Session 是状态变化的消费者（但不应直接依赖 Turn）
- Agent 作为胶水层，注册一个回调桥接两者
- 这就是 "Turn → Hook → Session 回调" 模式

**原则 3：无消费者的事件就是死代码**
- 如果没有任何 `hooks.on("xxx", ...)` 注册 `xxx` 事件，emit 它就是无意义的空转
- 除非注释说明这是"扩展点预留"，否则移除

### 7.2 数据流对比

**改前（持久化路径）**：
```
Agent.run()
  ├─ 正常结束：self._state_store.save_messages(...) + self._state_store.save_state(FINISHED, ...)
  ├─ 取消：    self._state_store.save_messages(...) + self._state_store.save_state(IDLE, ...)
  ├─ asyncio取消：self._state_store.save_messages(...) + self._state_store.save_state(IDLE, ...)
  └─ 异常：    self._state_store.save_messages(...)  ← 注意这里没存 state
```
问题：4 个分支各有硬编码的 StateStore 调用，且异常分支漏存 state。

**改后（持久化路径）**：
```
Agent.run()
  ├─ 正常结束：await self._persist(IDLE, {...})
  ├─ 取消：    await self._persist(IDLE, {...})
  ├─ asyncio取消：await self._persist(IDLE, {...})
  └─ 异常：    await self._persist(ERROR)
                    │
                    ▼
            Session.persist(messages, state, metadata)
              ├─ 更新 session.agent_state (内存)
              ├─ 更新 session.metadata (内存)
              └─ 调用 self._state_store.save_messages() + save_state()
```

**改前（状态同步路径 — Session 状态机是僵尸代码）**：
```
Turn emit state_change("thinking") → WebSocket 更新 UI
                                    → Session 完全不知情，始终是 IDLE
Agent.run() 结尾：
  self._state_store.save_state(session_id, FINISHED)
  → Session 只在首尾被更新
```

**改后（状态同步路径）**：
```
Turn emit state_change("thinking")
    ├─→ WebSocket hook → UI 更新
    └─→ Session hook → session.agent_state = THINKING

Agent.run() 结尾：
  await self._persist(IDLE)
  └─→ session.agent_state = IDLE (内存) + save_state (持久化)
```

### 7.4 状态机详解

**枚举定义**（state.py 最终形态）：
```python
class AgentState(str, Enum):
    IDLE = "idle"                    # 空闲。run() 未执行，或刚结束回到此状态
    THINKING = "thinking"            # LLM 推理阶段（对应 Anthropic extended thinking）
    GENERATING = "generating"        # LLM 流式输出中（首次 text_delta 后进入）
    WAITING_TOOL = "waiting_tool"    # 工具批量执行中
    WAITING_HITL = "waiting_hitl"    # 等待人工审批（Phase 2，当前仅预留）
    ERROR = "error"                  # 执行中出现未捕获异常
```

**状态转移规则**（编码 agent 实现 `_on_session_state_change` 时参考）：
```
IDLE         → (run开始) → THINKING
THINKING     → (首 chunk) → GENERATING
GENERATING   → (有 tool_calls) → WAITING_TOOL
GENERATING   → (无 tool_calls，循环结束) → IDLE
WAITING_TOOL → (工具全部完成) → THINKING (下一轮)
任意状态     → (异常) → ERROR
```

**重要**：ModelTurn 在 LLM 报错时会 emit `state_change("error")`（turns.py:224），此事件会被 Session 回调捕获，将 `session.agent_state` 设为 `ERROR`。但这只是瞬时状态——如果 Agent.run() 的 except 块重新调了 `_persist(IDLE)`，`IDLE` 会覆盖 `ERROR`。这是正确的行为：只有未被捕获的、导致 run() 终止的错误才应持久化为 ERROR。

async def persist(self, messages=None, state=None, metadata=None):
    """一站式持久化：更新内存状态并写入持久层。"""
    if state is not None:
        self.agent_state = state
    if metadata is not None:
        self.metadata.update(metadata)
    if messages is not None:
        await self.save(messages)
    else:
        await self.save()

    return await self._state_store.load_messages(self.id)

### 7.7 潜在坑点

**坑 1：`AgentState` 改名的影响范围**

`RUNNING` → `GENERATING` 不只是改枚举。需要全局搜索确认所有引用点：

```bash
grep -rn "RUNNING\|\.running\b" myagent/ --include="*.py"
grep -rn "FINISHED\|\.finished\b" myagent/ --include="*.py"
```

已知引用点：
- `turns.py:213`：`state="running"` → 改为 `state="generating"`
- `state.py`：枚举值本身
- WebSocket 前端：收到 `{"type": "state_change", "state": "running"}` 改为处理 `"generating"`
- scratch 测试脚本可能有硬编码

**坑 2：`_persist` 不再检查 `self._state_store is None`**

当前代码每个分支都有 `if self._state_store:` 守卫。`Session.persist()` 内部调用 `self.save()`，而 `save()` 内部已经检查了 `if self._state_store:`。所以无需在 agent.py 中再检查。如果 Session 没有 `state_store`，`persist()` 只更新内存状态，不报错。

**坑 3：`metadata` 合并策略**

`Session.persist()` 中对 metadata 用了 `.update()`，是合并而非替换。多次调用 `persist` 会累积：
```python
await persist(msg, IDLE, {"foo": "bar"})  # metadata = {"foo": "bar"}
await persist(msg, IDLE, {"baz": "qux"})  # metadata = {"foo": "bar", "baz": "qux"}
```
当前设计用 `update` 是因为各分支 metadata 正交（正常结束不设 cancelled，取消不设 stop_reason），不冲突。

**坑 4：audit.log_event 参数格式参差不齐**

删除 hook.emit 后，audit.log_event 调用保留不变。注意 `session_end` 的 audit 调用传的是 `{"exit_reason": ...}`，和 `session_start`/`turn_start`/`turn_end` 传 `ctx.snapshot()` 格式不同。这是现有行为，保持不动。

**坑 5：删除 import Session 时的准确性**

`from .session import Session, SessionManager` → `from .session import SessionManager`。确认 agent.py 中没有直接引用 `Session` 类（line 275 的 `session = await ...` 是变量名）。
